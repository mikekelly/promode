# Promode opinion register

This is the canonical, complete index of every opinion the promode corpus (the brief, all nineteen agent defs, the plugin commands, and the routed mechanics docs) instantiates — the single, evolvable home of the owner's preferences, whose evolution is what drives change in the corpus — auto-imported into every agent session in this repo as shared vocabulary. Every clause in a def or the brief must serve an item here (cut the clause or add the item — root `CLAUDE.md`); the register carries slug + one-line statement + homes, while the full opinionated text stays in the homes (one idea, one home — never fork prose into here). Slugs are stable citable IDs, extracted and ratified via `tasks/09-opinion-inventory.md` (2026-07-07); that inventory is this register's audit trail and coverage matrix.

**Home keys:** B = `plugins/promode/PROMODE_MAIN_AGENT.md` (`§` = its section) · SE ME EW HW FW CW GD CPO SPD CTO CMO SM CR DBG VER EM AA AUD CRF = `plugins/promode/agents/<agent>.md` (SE = senior-engineer, ME = mid-level-engineer — the two share the TDD **engineer body**; EW = elite-worker, HW = high-level-worker, FW = fast-worker, CW = cheap-worker — the four share one byte-identical **generic-worker body**; GD = gui-driver; CPO = chief-product-officer, SPD = senior-product-designer; CMO = chief-marketing-officer (inherit), SM = senior-marketer (opus/high) — the marketing lane, tasks 38–43; AUD = auditor, CRF = constraint-reinforcer). **`WK`** = the worker family (EW/HW/FW/CW) as a unit — a `WK` home means all four (they differ only in `model:`/`effort:` frontmatter). The old **PDE** key (product-design-expert) is **retired**: it split into CPO (claim-side / crucial calls) + SPD (execution), task 23. **FW's meaning changed**: it was the old TDD-bound "fast-worker" engineer; that engineer role is now `ME`, and `FW` names the sonnet/medium *generic worker* (roster restructure, `docs/decisions/2026-07-agent-roster-restructure.md`). · CM = root `CLAUDE.md` · W = wiki (`plugins/promode/docs/agent-knowledge-wiki.md`) · RB = `runbooks/sync-a-shared-principle.md` · routed docs by name (d2d = `plugins/promode/docs/discovery-to-determinism.md`, refcon = `plugins/promode/docs/reference-conformance.md`, gherkin = `plugins/promode/docs/gherkin-style.md`; the marketing routed docs, all `plugins/promode/docs/marketing-*.md`: mkt-offers, mkt-creative = marketing-creative-and-copy, mkt-paid = marketing-paid-ads, mkt-outbound, mkt-research = marketing-customer-research, mkt-aisearch = marketing-ai-search, mkt-distribution, mkt-planning, mkt-loops, mkt-council). The old **lookbook** key/doc (`design-system-lookbook.md`, with its `live-reload-server.md` companion) is **retired** — superseded by refcon (T23–T29; `docs/decisions/2026-07-reference-conformance.md`) · CMD-<name> = `plugins/promode/commands/<name>.md`.
**Markers:** default altitude is decision↑ for B/CM homes and mechanics↓ for def/doc/command homes (deviations marked). `v` = verbatim family (byte- or sentence-shared) · `c` = calibrated (deliberately refitted to role/pin) · `e` = enforcement home (reviewer/CI checks it). Rationale travels in every home by default (M3); `w:X` = canonical/fullest why lives in X; `w:—` = deliberately bare (accepted per M1). `⚙` = **harness-pinned: re-verify on any Claude Code change** (per CM's optimise-for-current-harness goal). `⌛` = **market-pinned: re-verify against the live platform/upstream, NOT on a Claude Code change** — platform/engine/benchmark behaviour that decays as the *market* (not the harness) moves; a distinct glyph from `⚙` because the re-verification *trigger* differs (a Meta algorithm change, an AI-engine index shift — not a harness release), so ⌛ items sit in their own Market-pinned block, not the Harness-pinned one. verified-as-of dates on ⌛ items are the source-repo state at mining (2026-07-15), and every quantitative claim in an ⌛ item is the source repo's practitioner claim, **unverified by us**. *conv* = convergent family: one stance, no shared wording across homes — invisible to grep, only this register keeps it coherent.

## Components (existence-as-opinion)

Each subagent, command, and routed mechanics doc is itself an opinion — the claim that this need is real and earns a dedicated component — so each entry is a fork decision point: a fork that rejects an entry's underlying opinions deletes or replaces the component rather than carrying dead weight. There are no skills to list: per M5, every capability lives on a non-voluntary surface. Entries cite, never duplicate: the brief's `<delegation-map>` stays canonical for routing, each def / command / routed doc for mechanics. Routed-doc conditional reads are imperatives, not guarantees (✓2 live-measured one miss) — dispatch briefs also cite the doc path, belt-and-braces (B§prompting-subagents).
