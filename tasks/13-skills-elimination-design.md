# Design draft: eliminate skills from the promode plugin (CTO)

## Brief
- **Orient** — the ratified new opinion (maintainer, 2026-07-07): *promode avoids skills altogether, in favour of dedicated agents or prompting shared across agents — skills require voluntary invocation, which is non-determinate and yields inconsistent behaviour.* Read: `plugins/promode/docs/opinion-register.md` (Components section — the 7 skills, their consumers and coordination edges), the brief (`PROMODE_MAIN_AGENT.md`, note chunk budgets: ~9.6k/7.3k/9.1k of 10k×3), all skill SKILL.md files, `tasks/07-invocation-tax.md` (prior research: which skills are dispatch targets vs NL-triggered), root `CLAUDE.md` (altitude + register doctrine).
- **Specify** — a **design + delegation-ready migration plan** (drafted for the main agent + maintainer to ratify; execute nothing beyond probes):
  1. **Per-skill disposition** for all 7 (promode-audit, discovery-to-determinism, design-system-lookbook, task-docs, handoff, recovering-subagents, reinforce-design-constraints): dedicated agent (JIT context economics preserved, invocation becomes deterministic dispatch via the delegation map) / shared prompting in existing defs (small, always-relevant mechanics) / plugin `commands/` (user-invoked flows like /handoff) / brief chunk (main-agent-only mechanics — weigh a 4th chunk: hooks.json registration procedure exists). For each: why that home, context-economics delta (what loads when, vs today's JIT skill), and what breaks if mis-homed.
  2. **Harness preconditions, probed live not assumed** (run scratch probes yourself or state exactly what must be verified before execution): (a) can a promode *subagent* use the Agent tool to fan out its own sub-assessors (decides whether promode-audit can become a dedicated agent, or its fan-out protocol must live in the brief)? (b) do plugin-shipped `commands/` work as the user-invoked replacement for /handoff (and /promode-audit's slash form)? (c) anything else your design leans on that the harness hasn't already verified (CLAUDE.md records what has been).
  3. **Corpus consequences**: register changes (the new opinion as a row; Components skill entries → their new homes; description-tax rows retired), README (Skills section, Fork it references), CLAUDE.md (altitude paragraph currently names skills as a mechanics home), wiki authoring conventions, sync runbook, check scripts (skill-frontmatter check retires?), delegation map rewiring.
  4. **Sequencing** as delegable tasks (sized per agent, dependencies named, verification checkpoints placed) + the risks/one-way doors (e.g. NL triggers lost: today a user saying "audit this repo" fires the skill description; after migration the main agent must route it — is the delegation map sufficient?), and the strongest rejected alternatives with why.
- **Why** — voluntary invocation is the last non-deterministic delivery surface in promode; everything else reaches agents by guaranteed load (hook, defs, @-import) or dispatch. The maintainer judges the inconsistency cost above the JIT benefit; the design must preserve the JIT *economics* through dedicated agents while removing the *voluntariness*.
- **Verified vs assumed** — task 07 verified skills' invocation semantics on 2.1.201; subagent-spawning-subagents and plugin commands/ are UNVERIFIED assumptions until probed; chunk budgets verified 2026-07-07.
- **Not / exit** — design + probes only: no corpus edits, no version bump. Record the draft (or a link to it) + probe results in this doc's Outcome. Exit: a plan the main agent can ratify and dispatch task-by-task.

### 1. Harness preconditions — probed live on 2.1.201 (scratch plugin `probeplug`, scratchpad; all markers round-tripped)

| # | Question | Result |
|---|---|---|
| P1 | Can a plugin **subagent** use the Agent tool to fan out its own sub-agents? | **YES.** A plugin-defined subagent's tool set includes `Agent` (reported: Agent, Bash, Edit, Read, Skill, ToolSearch, Write, …); it launched a second plugin agent and returned its reply verbatim. Nesting main → plugin-agent → plugin-agent works. |
| P2 | Does **background** fan-out work *inside* a subagent? | **YES, with a routing wrinkle.** The parent spawned two `run_in_background` children in one message, ended its turn, was woken by notifications, and synthesised both results. BUT the children's task-notifications **also bubbled to the root session** (duplicated, once per child run/resume), and the root-session model reported having to relay results. Fan-out inside a dedicated agent should therefore use **foreground parallel batch** (multiple Agent calls in one message — verified clean in P1's foreground path), not fire-and-forget; O4 stays a *main-agent-only* protocol. |
| P3 | Do plugin `commands/` work as user-invoked flows? | **YES.** `claude --plugin-dir … -p "/probeplug:ping hello"` executed the command; `$ARGUMENTS` expanded (`arg=[hello]`); `${CLAUDE_PLUGIN_ROOT}` expanded **in the command body**; the command read a plugin-shipped file. |
| P4 | Are plugin commands model-visible/invocable (context tax)? | **YES.** The command was invocable by name via the Skill tool in a fresh session — commands and skills share the merged listing + Skill-tool machinery on 2.1.201. So `commands/` does **not** remove the description tax or model-invocability by itself; it changes *convention*, and `disable-model-invocation` remains the lever if we ever want typed-only (task 07 verified it for plugin skills on this version; assumed same for commands — not separately probed). |
| P5 | Does `${CLAUDE_PLUGIN_ROOT}` expand in plugin **agent defs**? | **YES.** The leaf agent's def referenced `${CLAUDE_PLUGIN_ROOT}/marker.txt`; the path arrived expanded and the file was read. This is the load-bearing mechanism for def-directed doc reads and the inspector script path. |

All five are **harness-pinned (⚙)** — add to the register's re-verify list with version 2.1.201 / 2026-07-07. Incidental P4 observation: a command body written as bare imperatives ("Do exactly this…") was flagged as injection-shaped by a fresh sonnet when loaded via the Skill tool mid-conversation; command bodies should be phrased as task instructions for the invoked flow, as `handoff` already is.

### 2. Design — the delivery-surface matrix that replaces skills

Skills die; every capability moves to one of four **non-voluntary** surfaces (this is the new opinion's operational form):

1. **Dedicated agent** — big JIT mechanics that are *one agent's whole job*; invocation becomes delegation-map dispatch. Preserves JIT economics exactly: the def is the agent's system prompt, loaded only when dispatched.
2. **Shared/def prompting** — small, always-relevant-to-that-agent mechanics, inline in the def (register-tracked homes).
3. **Def-directed doc read** — *cross-cutting* mechanics several agents need occasionally: content at `plugins/promode/docs/<name>.md`, each consuming def carrying a conditional imperative ("when the dispatch is X, first read `${CLAUDE_PLUGIN_ROOT}/docs/<name>.md`", P5-verified). Not literally deterministic (still model compliance), but it swaps *listing-competition voluntariness* for an *in-system-prompt imperative* — the same trust level the corpus already grants def text ("dispatch is acted on").
4. **Plugin `commands/`** — user-typed flows. Keep model-invocable so the brief can also direct them (P4 means they stay listed; the residual description tax is bought deliberately).

Main-agent-only mechanics go in the **brief** (chunk 4 — registration procedure exists, budget verified below).

### 7. Risks, one-way doors, reversibility

- **Reversible (cheap):** everything in-repo — git revert + patch release restores skills; no external state. Slash names are preserved (`/promode:handoff`, `/promode:promode-audit`), so no user muscle-memory breakage. This migration is **not** a one-way door in the `disable-model-invocation` sense (task 07's flag *was* one; we're not using it).
- **Harness-pinned (the real risk):** the design now leans on P1/P2/P5 (nested Agent fan-out; `${CLAUDE_PLUGIN_ROOT}` in defs/commands) — undocumented-or-lightly-documented behaviours that could regress in a future Claude Code release. Mitigation: register ⚙ rows + the existing re-verify discipline; the auditor def should state its fallback ("if the Agent tool is absent, report the dimensions + rubric back for the main agent to fan out" — degraded, not broken).
- **NL-trigger loss is net-positive, with one soft spot:** "audit this repo" now routes via the *guaranteed-loaded* brief instead of listing luck — stronger. The soft spot is subagent-side triggering: a def's conditional read fires only when the def's condition matches the dispatch; a *mis-briefed* dispatch ("fix the flaky E2E suite" sent to FW without naming GUI work) previously might have skill-triggered off d2d's description. Judged acceptable: promode's own doctrine says routing is the main agent's job, and O8-quality briefs name the work.
- **P2 wrinkle:** background-notification bubbling to the root session is undocumented behaviour observed once; the foreground-batch protocol avoids it entirely, but note it in the auditor def's why.
- **Chunk-4 tail-drop:** mis-registration silently drops it — covered by existing CI check; ✓1 verifies live.

### 8. Strongest rejected alternatives

1. **`disable-model-invocation` on qualifying skills** (task 07's lever) — rejected there and still: 6/7 are dispatch targets; the flag is a plugin-author one-way door; buys tokens, not determinism.
2. **d2d/lookbook as dedicated agents** — the closest call. Rejected because the knowledge is *cross-cutting* (SE builds, VER runs, FW drives, PDE bridges): an agent home can't deliver mechanics to sibling agents, and a seam-engineer def would fork the checksummed TDD blocks (RB sync burden, drift risk). Def-directed doc reads keep one home and exact JIT economics.
3. **Everything into the brief** — dies on the 10k×N budget (audit alone is 23k) and puts subagent mechanics at the wrong altitude (main agent doesn't build seams).
4. **Auditor keeps synthesis in the main agent** (today's shape, agent only gathers) — rejected: the rubrics would then still need a main-agent-loadable home (the eliminated skill), and P1 removes the technical reason the fan-out ever lived in the main context. The O2 ratify pattern covers the judgement concern.
5. **A `docs/`-only migration (no new agents; every skill becomes a def-directed doc)** — uniform, but audit and constraint-reinforcement are *whole jobs*, not reference knowledge; routing them deserves delegation-map determinism, and the maintainer's ratified opinion names dedicated agents as the preferred home.

**Assumptions (challenge before ratifying):** chunk 1–3 edits stay size-neutral (re-run `check-hook-output-limits.sh` during C — budgets have ~0.4–2.7k headroom); `disable-model-invocation` semantics for *commands* were not probed (unneeded by this design — flagged in case ratification wants typed-only handoff); P2's notification-bubbling was observed once, not characterised across versions; token estimates are chars/4 approximations.
